## Cross-Phase Integration

### Bug 1: extract_metadata ignores OllamaProvider (CRITICAL)

`ai_extractor.extract_metadata()` accepts `provider` and `fallback_provider` parameters but never reads them. The function always routes through `_try_model` → `_create_client`, which builds an `openai.OpenAI` client pointed at ZAI/OpenRouter base URLs. There is no branch for `active_provider == "ollama"`.

**Impact:** When user selects local provider:
- OllamaProvider.complete() is never called
- llama-server at localhost:8080 never receives requests
- Documents are processed through ZAI/OpenRouter, not the local model

**Affected requirements:** SRVR-01, SRVR-02, PROV-01, PROC-01

### Bug 2: sanitize_metadata called incorrectly (CRITICAL)

`sanitize_metadata(raw: dict) -> dict` expects and returns a plain dict. At call sites (lines 309, 320), it's called with a `ContractMetadata` dataclass instance, and the return value is discarded.

**Impact:** Even if fixed, post-processing output is thrown away — unsanitized data goes downstream.

**Affected requirements:** PROC-01

## Root Cause

Both bugs stem from the same architectural gap: `ai_extractor.py` has a legacy code path (`_try_model` → `_create_client`) that pre-dates the provider abstraction. The `provider` parameter was added to the function signature (Phase 1, v0.4) but the actual extraction logic was never refactored to use it. Phase 5 added `sanitize_metadata` calls inside `extract_metadata`, but since the function never routes to OllamaProvider, the calls are effectively dead code.

**Fix:** Refactor `extract_metadata` to route through `provider.complete()` when a provider is passed, instead of always using `_try_model`. Then fix `sanitize_metadata` call to convert ContractMetadata → dict → ContractMetadata.
