# Requirements: ЮрТэг — Веха 1 (Бэкенд до продакшена)

**Defined:** 2026-03-19
**Core Value:** Юрист загружает папку с документами и за 20 минут получает готовый реестр — без ручного ввода, без обучения, без «проекта внедрения». Всё что можно автоматизировать — автоматизировано.

## v1 Requirements

### Фундамент

- [x] **FUND-01**: Обновление приложения не ломает существующую базу данных пользователя — версионированные миграции схемы
- [x] **FUND-02**: Бизнес-логика обработки документов отделена от интерфейса и может вызываться независимо (сервис-слой)
- [x] **FUND-03**: AI-провайдер переключается через конфигурацию (GLM / OpenRouter / будущая Ollama) без изменения кода
- [x] **FUND-04**: Даты, извлечённые AI, нормализуются и проверяются на корректность перед сохранением

### Жизненный цикл документа

- [x] **LIFE-01**: Документ автоматически получает статус (действует / скоро истекает / истёк) на основе даты окончания — вычисляется на лету, всегда актуален
- [x] **LIFE-02**: Юрист может вручную установить статус (расторгнут / продлён / на согласовании / приостановлен) — ручной статус приоритетнее автоматического
- [x] **LIFE-03**: При загрузке обновлённого договора система автоматически распознаёт что это новая версия существующего документа (по эмбеддингам + метаданным файла: дата создания, автор) и связывает их в единую карточку с таймлайном версий
- [x] **LIFE-04**: Юрист может посмотреть что изменилось между версиями документа (суммы, даты, контрагенты, ключевые условия) + генерация redline в .docx формате (track changes для Word)
- [x] **LIFE-05**: При открытии приложения сразу видна панель «требует внимания» — документы с истекающими сроками и проблемами
- [x] **LIFE-06**: Пороги предупреждения о сроках настраиваются (30 / 60 / 90 дней) глобально
- [x] **LIFE-07**: Платёжный календарь — визуальный график всех платежей по договорам (кому платить, от кого получить, когда), на основе извлечённых AI сумм и дат
- [x] **LIFE-08**: AI-ревью договора против шаблона — библиотека эталонов по типам договоров (NDA, поставка, аренда и т.д.), автоматическое сопоставление нового договора с нужным шаблоном, показ всех отступлений (через векторные эмбеддинги для точности, опционально)

### Профили и мультидоступ

- [ ] **PROF-01**: Один юрист может вести документы нескольких компаний/клиентов — реестр разделён, можно переключаться между клиентами
- [x] **PROF-02**: Несколько юристов могут работать с общим реестром одной компании — видят одни и те же документы и статусы

### Интеграции

- [ ] **INTG-01**: Telegram-бот принимает документы (PDF/DOCX) — кинул файл в чат, через минуту он в реестре. Без дополнительных действий
- [ ] **INTG-02**: Telegram-бот отправляет уведомления о приближающихся сроках и проблемных документах
- [x] **INTG-03**: Подключение папки Google Drive — новые файлы обрабатываются автоматически при появлении
- [ ] **INTG-04**: Уведомления внутри приложения (всплывающее при запуске) о документах, требующих внимания

### Безопасность и доверие

- [ ] **SECR-01**: Журнал действий — лог всех операций с документами (что обработано, когда, какой AI-провайдер использован) для аудита
- [ ] **SECR-02**: Приложение упаковано в Docker — B2B-клиент запускает на своём сервере одной командой

## v2 Requirements

### Расширенные интеграции

- **INTG-05**: API для интеграции со сторонними системами (Notion, CRM)
- **INTG-06**: Excel/CSV экспорт реестра в произвольном формате
- **INTG-07**: Яндекс.Диск — аналогично Google Drive

### Расширенное версионирование

- **LIFE-10**: Визуальный diff между версиями документа (подсветка изменённых абзацев)
- **LIFE-11**: Автоматическое извлечение условий автопролонгации из текста договора

### Команда

- **PROF-03**: Роли и права доступа (администратор / юрист / просмотр)
- **PROF-04**: Партнёрский доступ для аутсорсеров (одна подписка — несколько клиентов)

## Out of Scope

| Feature | Reason |
|---------|--------|
| Режим «только локально» | Приложение всегда работает локально в продакшене |
| Email-уведомления | Telegram покрывает потребность, email требует SMTP-инфраструктуру |
| Мобильное приложение | Web-first, mobile потом |
| Облачная синхронизация | Противоречит local-first позиционированию |
| Импорт из WhatsApp | Нет API для автоматизации |
| Кнопка переключения AI-моделей в интерфейсе | Переключение через конфиг, пользователь не должен об этом думать |
| A/B тестирование промптов | Веха 3 (LLM) |
| Fine-tuning pipeline | Веха 3 (LLM) |
