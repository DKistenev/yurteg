# ЮрТэг — Канонический workflow для людей и AI-агентов

> Этот файл — основной источник истины по совместной работе в репозитории.
> Claude-specific детали держим в `CLAUDE.md`, без дублирования общих правил.

## Проект

ЮрТэг — desktop-приложение на NiceGUI для AI-обработки юридических документов.
Основатель: Daniel (Даниил), юрист, без выделенной команды разработчиков.

## Язык работы

- Русский для общения, комментариев в коде, commit message и PR.

## Быстрый старт

```bash
# Приложение
PYTHONPATH=. python app/main.py

# Тесты
pytest
pytest -m "not slow"

# Telegram-бот
python -m bot_server.main
```

## Базовые правила репозитория

- `main` не редактируем напрямую: любая содержательная работа идёт из короткой ветки.
- Merge только через PR, даже для маленьких изменений.
- После merge рабочую ветку удаляем локально и на GitHub.
- `.planning/` — часть проекта и остаётся в git.
- Секреты, токены, локальные runtime-файлы и tool logs не коммитим.

### Именование веток

Используем короткие осмысленные префиксы:

- `codex/<task>`
- `claude/<task>`
- `feat/<task>`
- `fix/<task>`
- `docs/<task>`
- `chore/<task>`

## Координация агентов

Если над проектом одновременно работают Codex, Claude Code или другой агент:

- один агент не редактирует файл, который уже взял другой, без явного согласования;
- перед работой агент объявляет, над чем работает;
- по завершении агент пишет, что именно закончил;
- если сервер координации недоступен, это явно фиксируется в сообщении/PR, а не замалчивается.

### Agent-MCP

Проект поддерживает координацию через `agent-mcp` (`http://localhost:8888`), если он поднят локально.

Ожидаемый протокол:

1. В начале сессии прочитать текущий project context и открытые задачи.
2. Обновить свой статус в project context.
3. Перед редактированием проверить статус файла и, если сервер доступен, пометить файл как занятый.
4. После завершения снять блокировку и обновить `last_completed`.

### Секреты Agent-MCP

- Admin token и любые другие секреты не храним в git.
- Токены держим в локальном секрете, локальном MCP-конфиге или переменных окружения.
- Если агенту нужен token, он должен быть получен из локальной среды, а не из репозиторного файла.

## Владение файлами

- Не трогаем чужие активные файлы без договорённости.
- Если задача широкая, разбиваем работу по подсистемам или файлам.
- Коммиты держим атомарными: один логический смысл на один commit.
- В больших изменениях указываем `Co-Authored-By`, если работа действительно совместная.

## Что считаем локальным шумом

В git не должны попадать:

- каталоги рантайма агентов: `.agent/`, `.agents/`, `.claude/worktrees/`;
- логи и артефакты инструментов: `.playwright-mcp/`, `.superpowers/`, `.understand-anything/`;
- локальные IDE/MCP-конфиги: `.vscode/`, `.mcp.json` и подобные;
- временные скриншоты в корне репозитория: `audit-*`, `debug-*`, `final*`, `screenshot-*` и похожие;
- локальные skill bundles и одноразовые заметки.

Если артефакт полезен проекту, его нужно:

1. либо положить в осмысленное место внутри документации;
2. либо явно обсудить и только потом трекать в git.

## Что остаётся в репозитории

- исходный код приложения и бота;
- тесты;
- `.planning/` как часть проектного процесса;
- осмысленная документация;
- скрипты и конфиги в `dataset/`, если они реально нужны проекту.

`dataset/` считаем частью проекта, но generated training outputs и тяжёлые промежуточные файлы должны быть ignored.

## GitHub workflow

Минимальный рабочий процесс:

1. Создать короткую ветку от актуальной базы.
2. Сделать локальные изменения и прогнать релевантные проверки.
3. Открыть PR с понятным summary и коротким чеклистом.
4. Не смешивать в одном PR feature, cleanup и несвязанные правки.
5. После merge удалить ветку.

Если в репозитории уже есть живой PR, новую тему выносим в отдельную ветку, а не подмешиваем в старый PR.

## PR expectations

В PR должно быть понятно:

- что меняется;
- зачем это нужно;
- какие проверки выполнены;
- есть ли риски, долги или follow-up.

## Технические инварианты проекта

- Все пути через `pathlib.Path`, а не `os.path`.
- Исходные документы только копируются, не перемещаются и не удаляются.
- Анонимизированный текст не сохраняется на диск.
- Падение на одном файле не должно валить весь архив.
- UI-стили идут через design tokens в `app/styles.py`, без хардкода.
